# 云摊坊待开发清单

本文档记录后续开发必须覆盖的核心内容、核心链路和验收标准。后续 AI 或开发者接手时，优先阅读本文档，再按需查看 `docs/frontend-page-gap-analysis.md`、`docs/module-roadmap.md`、`docs/agent-skills/TODO.md`。

## 优先级说明

- P0：核心闭环或当前验收阻塞，必须优先完成。
- P1：核心能力增强，影响完整可用性。
- P2：非核心体验、运营、工程化优化，可在主链路稳定后推进。

## P0 当前验收链路

### 1. 用户端首页摊位释放验证

- H5 首页“附近摊位”当前只应展示已释放/可经营的摊位。
- 在后台未审核释放前，用户端首页、摊位列表、搜索结果都不能展示“老城豆花摊”。
- 初始可见摊位应包含“烟火小摊”“乡野新农人鲜铺”。
- 搜索“老城豆花摊”时，如果摊位未释放，必须搜不到。
- 摊位可见状态必须由真实后端字段控制，不能前端写死隐藏。

### 2. 商家端待审状态验证

- H5 使用 `test4 / 123456` 登录后，应进入商家端。
- 登录页或快捷按钮需要提供“待审核摊主”入口，便于测试。
- 商家工作台必须展示真实的入驻与认证状态，当前应为待审核。
- 商家资质、特殊身份/公益标签、摊位预约页面必须读取真实待审数据。
- 这些待审数据不能使用前端 mock；刷新页面后状态应保持一致。

### 3. 管理后台四级审核

- 管理后台使用 `test3 / 123456` 登录。
- 入驻审核：通过“老城豆花摊”后，商家基础入驻状态变为已通过。
- 资质审核：通过资质后，商家认证状态更新；如果入驻未通过，需要阻止或提示不能审核资质。
- 特殊身份/公益标签审核：通过后，前台商家资料展示公益/特殊标签。
- 摊位预约审批：通过后，该摊位被释放到用户端首页、列表和搜索。
- 每个审核菜单都必须使用真实数据，并提供通过/驳回按钮。
- 驳回必须记录驳回原因，并回写到商家端对应页面。

### 4. 回到用户端验证释放结果

- 完成后台四级审核后，刷新 H5 首页，“老城豆花摊”应出现。
- 用户可以搜索到“老城豆花摊”。
- 用户可以进入“老城豆花摊”详情页。
- 用户可以看到该摊位真实商品、真实价格、真实摊位信息。
- 用户可以从该摊位发起预约下单。
- 订单必须写入真实订单表，订单关联真实用户、真实商家、真实摊位、真实商品和真实价格。
- 下单后用户端“我的订单”可见，商家端“订单处理”可见。

## P0 用户端核心闭环

- 登录态：未登录默认进入 `/login`；登录后按角色跳转用户端、商家端或管理后台。
- 角色隔离：普通用户不能访问 `/vendor/*` 和管理后台；商家不能进入普通用户专属操作；管理后台仅 admin 可访问。
- 首页：展示已释放摊位、分类筛选、推荐流、搜索入口，分类筛选必须在当前页面生效。
- 摊位列表：支持关键词、分类、距离、评分、营业状态筛选；只展示已释放摊位。
- 摊位详情：展示摊位信息、摊主信息、营业状态、地址、评分、评价、商品列表、收藏、关注、订阅入口。
- 商品数据：不同摊位必须有不同商品；商品需包含名称、价格、库存/可售状态、分类、图片或占位图。
- 预约下单：必须明确展示摊位、商品名、数量、单价、总价、取货时间、备注、联系方式。
- 订单创建：订单必须落库到真实订单表和订单明细表。
- 我的订单：订单列表必须展示商品名、数量、单价、总价、取货时间、订单状态。
- 订单详情：展示订单完整信息、订单明细、摊位、商家、状态流转、评价/投诉入口。
- 订单状态：支持待接单、备货中、待取餐、已完成、已取消、已拒单等状态展示。
- 评价：已完成订单可评价；评价必须关联订单、摊位、商家和用户。
- 投诉：用户可对订单或摊位提交投诉；投诉必须进入管理后台工单。
- 收藏关注：收藏摊位、关注摊主、订阅出摊提醒必须持久化。
- 我的页：常用入口应进入真实页面或明确的待开发占位，不能出现无意义小字或不可点击入口。

## P0 商家端核心闭环

- 入驻申请：商家可提交摊主名称、简介、联系方式、经营品类等基础信息。
- 商家工作台：显示入驻状态、资质状态、特殊身份/公益标签状态、摊位预约状态、订单概览。
- 资质管理：支持提交健康证、摊车照片、经营材料等资质记录；支持查看审核状态和驳回原因。
- 特殊身份/公益标签：支持提交身份或公益标签申请；通过后在用户端展示标签。
- 摊位预约：商家可选择可申请摊位，提交预约申请，查看审核结果。
- 摊位释放：后台审批通过后，商家摊位进入可经营/可展示状态。
- 商品管理：支持新增、编辑、上下架商品；商品必须绑定商家和摊位。
- 商品展示：商家商品更新后，用户端摊位详情必须同步展示。
- 订单处理：商家端订单列表必须显示商品名、数量、单价、总价、取货时间、下单用户信息和状态。
- 状态流转：商家可接单、拒单、标记备货中、完成订单；状态更新后用户端同步。
- 评价回复：商家可查看评价并回复，回复内容在用户端可见。
- 摊主故事：支持发布、编辑、下架故事；内容进入用户端故事流。

## P0 管理后台核心闭环

- 登录与权限：管理后台独立登录，普通用户和商家不能访问。
- 数据总览：使用真实统计数据展示用户数、商家数、摊位数、订单数、待审核数、投诉数。
- 入驻审核：查看商家申请详情，支持通过/驳回，驳回原因回写商家端。
- 资质审核：查看资质材料，支持通过/驳回，和商家认证状态联动。
- 特殊身份/公益标签审核：支持通过/驳回，通过后用户端展示。
- 经营区域管理：支持区域新增、编辑、启停；后续接地图围栏。
- 摊位管理：支持摊位新增、编辑、启停、释放状态管理。
- 摊位预约审批：支持通过/驳回；通过后释放摊位到用户端。
- 投诉工单：支持查看、分派、处理、关闭投诉；处理状态回写用户端。
- 内容审核：支持审核摊主故事、评价图片等内容。
- 违规记录与信用分：投诉、违规、审核结果要能影响商家信用分。
- 政策公告与通知：后台可创建、编辑、发布政策公告，用户端可查看。
- 用户与角色：支持查看账号、角色、状态；后续支持禁用与权限配置。

## P0 Agent 核心闭环

- Agent 入口：用户端左侧导航“快速开始”进入 Agent 页面。
- Agent 会话：切换页面后会话应保留一定时间；右侧保留会话列表。
- 参数校验：每个 skill 必须明确必填参数，缺少必填参数时必须追问，禁止模型脑补。
- 搜索摊位：调用 API 后必须把返回结果处理成可读文本和卡片，包含摊位名、分类、状态、距离、评分、地址。
- 下单：必须要求明确摊位、商品、数量、取货时间；下单后创建真实订单，用户端和商家端均可见。
- 评价：必须要求明确订单和评分；缺少订单时引导用户选择订单。
- 投诉：必须要求明确投诉对象、类型和描述；提交后进入管理后台投诉工单。
- 帮助说明：可回答系统入口在哪、某功能如何使用、当前能做什么。
- 处理过程：右侧可展示流动式步骤，体现识别意图、检查参数、调用 API、返回结果。
- 失败重试：API 失败后最多重试 3 次；超过 3 次直接提示功能暂时不可用。
- Skill 文档：`docs/agent-skills/SKILL.md` 和子 skill 必须与真实 API 参数保持同步。

## P1 数据与接口补齐

- 补齐数据库迁移：账号、角色、商家、资质、特殊身份、摊位预约、摊位释放、商品、订单、评价、投诉、收藏、订阅、通知。
- 补齐测试账号种子数据：`test1` 普通用户、`test2` 已审核商家、`test3` 管理员、`test4` 待审核商家。
- 明确摊位可见规则：只有已释放、启用、审核通过、当前可经营的摊位进入用户端查询。
- 明确订单金额计算：以后端商品单价为准，前端传入金额仅作展示或校验。
- 订单明细接口统一返回商品名、数量、单价、小计。
- 商品接口支持按摊位查询，避免不同摊位商品混在一起。
- 后端所有写接口增加参数校验和清晰错误信息。
- 前端 API 类型从 `any` 逐步改为明确接口。
- 统一分页结构、状态枚举、时间格式和金额格式。

## P1 前端页面补齐

- 用户端摊主主页：展示摊主简介、荣誉标签、公益标签、故事流、关注按钮。
- 用户端评价列表：支持按最新、评分、图片评价筛选。
- 用户端消息中心：接收订单状态、审核结果、投诉处理、出摊提醒。
- 用户端钱包：订单支付、退款、优惠券、账单记录。
- 用户端设置：账号安全、隐私、退出登录。
- 商家端商品编辑页：图片、价格、库存、分类、上下架、所属摊位。
- 商家端订单详情页：订单明细、用户信息、操作记录、状态按钮。
- 商家端摊位申请详情页：摊位位置、时段、规则、审核记录。
- 商家端荣誉/信用页：信用分、违规记录、官方标签。
- 管理后台各菜单从通用 `WorkbenchView` 拆成真实业务页。
- 管理后台审核详情页：展示完整申请材料和审核历史。
- 管理后台报表页：摊位使用率、订单量、投诉率、商家信用分分布、导出。

## P1 测试与验收

- 写端到端验收脚本：登录 test4 提交/查看待审，登录 test3 四级审核，登录 test1 搜索并下单。
- 写 API 集成测试：商家入驻、资质审核、公益标签审核、摊位预约审批、订单创建、商家接单。
- 写 Agent 集成测试：缺参数追问、搜索摊位返回卡片、下单创建真实订单。
- 每次合并后至少运行 H5 构建、管理端构建、后端编译。
- 后端应加入 Maven wrapper 或在文档中明确 Maven 安装要求。

## P2 非核心增强

- 地图真实接口：摊位位置、导航、区域围栏、人流热力。
- 支付真实接口：支付、退款、支付状态轮询、取餐码。
- 上传能力：资质图片、商品图片、故事图片、投诉证据。
- 内容安全：评价、故事、投诉内容审核。
- 推荐算法：根据距离、口碑、等待时间、用户偏好在首页推流，不做独立入口。
- 数据埋点：曝光、点击、搜索、下单转化、Agent action 成功率。
- 运维监控：后端日志、慢接口、异常告警、数据库迁移记录。
- UI 细节：移动端手势、PC 响应式密度、空状态、加载态、错误态。

## 当前技术债

- 管理后台大量菜单仍复用 `WorkbenchView`，真实业务交互需要逐步拆分。
- 部分用户端工具页仍是通用占位页，需要变成独立页面。
- Agent skill 与真实 API 需要持续同步，避免参数漂移。
- 前端部分 API 返回值仍使用 `any`，类型约束不足。
- 当前环境没有 Maven wrapper，后端编译依赖本机 Maven。
- 需要避免新增 mock 数据掩盖真实数据库状态，验收链路必须基于真实接口。
