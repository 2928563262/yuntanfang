# submit_complaint

## 功能

帮助用户把自然语言投诉整理为投诉工单。

## 触发语

- "我要投诉烟火小摊"
- "摊位卫生有问题"
- "有人占道经营"

## 参数

```json
{
  "target": "烟火小摊",
  "type": "卫生问题",
  "description": "现场卫生需要核实。"
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `target` | string | 必填 | 投诉对象，摊位、摊区或订单 |
| `type` | string | 条件必填 | 问题类型，如卫生问题、占道经营、商品质量、服务问题 |
| `description` | string | 条件必填 | 详细描述 |

必须有 `target`，且 `type` 和 `description` 至少需要一个。只说“我要投诉”时必须先追问对象；只说“投诉烟火小摊”时必须追问具体问题。

系统默认值：

- 用户提供对象和描述但未提供类型时，后端使用 `其他问题`。

禁止行为：

- 不要把“投诉卫生问题”默认绑定到某个摊位。
- 不要把“我要投诉”直接生成投诉工单。
- 不要根据最近浏览摊位默认投诉对象，除非用户明确说“这个摊位”且上下文已经确认。

## 参数不足响应

追问文案：

缺投诉对象：

`你要投诉哪个摊位、订单或摊区？请先告诉我投诉对象。`

缺问题描述：

`你要投诉的具体问题是什么？可以说卫生问题、占道经营、商品质量或服务态度。`

## 响应 action

```json
{
  "type": "submit_complaint",
  "label": "确认提交投诉",
  "route": "/complaints/create",
  "payload": {
    "target": "烟火小摊",
    "vendorId": 1,
    "stallId": 1,
    "type": "卫生问题",
    "description": "现场卫生需要核实。",
    "status": "待提交"
  }
}
```

## 前端行为

用户确认后调用 `POST /api/complaints`。若后端能通过 `target` 匹配真实摊位，payload 必须带上 `vendorId` 和 `stallId`；若用户明确说了订单号，也要带上 `orderId`。成功后跳转 `/complaints`。
